# SECTION 1: MASTER BLUEPRINT FOR "AXIOM"
### MCP Protocol Observability & Security Platform

## 1.1 The Ultimate Thesis

**What AXIOM is.** AXIOM is the telemetry, proxy, and perimeter security layer for systems built on the Model Context Protocol (MCP) — the protocol Anthropic introduced in November 2024 that standardizes how AI agents discover and call external tools, and which has since become the closest thing the agentic-AI world has to a universal interface, often described as "USB-C for AI." AXIOM sits transparently inside the standard MCP chain — User ↔ MCP Host ↔ MCP Client ↔ MCP Server(s) ↔ Tools/Data/APIs — intercepting every message in both directions without requiring the agent framework or the MCP server to know it's there.

**Why now, precisely.** MCP adoption is not a slow curve — FastMCP, just one of the popular server-building frameworks, is downloaded over a million times a day. Every major agent client (Claude Desktop, Cursor, Claude Code, Gemini CLI, Windsurf, ChatGPT) now speaks it. And every one of those integrations inherits the same structural problem: traditional observability tooling — Datadog, OpenTelemetry, Postman — was built for a world where the developer controls the schema and the API surface is fixed at compile time. MCP inverts that. The agent discovers tools at runtime, from any server it's been pointed at, and a third party — the server author, not the application developer — controls what context and instructions the model actually sees. Two servers connected to the same agent share the same flat context window, meaning a malicious server's tool description can rewrite how the agent treats a completely different, trusted server's tools, without ever being called directly. None of that has a REST or gRPC analog. A tool built for stateless, developer-defined API calls has no vocabulary for a stateful, third-party-defined, natural-language-interpreted handshake — it is structurally blind to the entire category of failure.

**The core failure modes, with real evidence behind each one.** This isn't theoretical risk-modeling. Between January and February 2026 alone, security researchers filed more than 30 CVEs against MCP servers, clients, and infrastructure, 43% of them shell-injection-class vulnerabilities. CVE-2025-6514, found in the widely used `mcp-remote` package, scored a 9.6 on CVSS and had already been downloaded over 437,000 times before disclosure. Palo Alto Networks' Unit 42 measured a 78.3% attack success rate on setups where just five MCP servers were connected to a single agent. And this isn't confined to obscure, unmaintained tooling — January 2026 disclosures included path traversal and argument-injection vulnerabilities in Anthropic's own official Git MCP server. The specific failure modes AXIOM exists to close:

- **Tool poisoning** — Invariant Labs' original disclosure showed a tool innocuously named "add" with a description reading "Add two numbers" that, appended invisibly, instructed the model to read `~/.ssh/id_rsa` and pass its contents as a parameter. Because tool descriptions are injected directly into the model's context and treated as trusted instructions, this is both trivial to execute and easy to miss without inspection at the message level.
- **Rug-pull attacks** — a server's tool descriptions can change after a user has already approved them once, and nothing in the protocol re-prompts for review. A tool that was safe on Monday can be malicious by Thursday with no version bump and nothing for a package manager to flag.
- **Confused deputy and over-scoped tokens** — MCP servers frequently execute with their own broad privileges rather than the requesting user's actual permissions, and routinely request wider OAuth scopes than the task needs (full mailbox access instead of read-only, for instance), turning an ordinary integration into an aggregation risk.
- **Silent multi-agent chain failures and shadow servers** — unsanctioned MCP instances spun up by individual developers for convenience, running outside any formal security review, are the MCP-era version of Shadow IT, and they routinely ship with default credentials or permissive configurations.
- **Data exfiltration through legitimate-looking channels** — sensitive data encoded into ordinary-seeming tool calls (a search query, an email subject line) rather than moved through any channel a conventional DLP tool would flag.

## 1.2 The Complete Technical Architecture

**Architecture topology.** An AI agent framework (LangGraph, CrewAI, AutoGen, or a direct Claude/GPT agent loop) is configured to route all MCP traffic through the AXIOM proxy rather than connecting directly to each MCP server. The proxy terminates the agent-facing connection, forwards (or blocks, or rewrites) traffic to the real MCP server(s), and mirrors every frame to AXIOM's analysis and storage layer — a design deliberately analogous to how an API gateway or service mesh sidecar sits in front of conventional microservices, adapted for a protocol where the "requests" are JSON-RPC calls a model chose to make rather than calls a developer coded.

**Infrastructure and backend stack.** The backend runs on FastAPI specifically for its native async support — MCP's `stdio` and Server-Sent Events (SSE) transports are both long-lived, streaming connections, and a synchronous framework would serialize what needs to stay concurrent across potentially dozens of simultaneous agent sessions. Each customer's proxy instance runs in an isolated, modular Docker container, giving multi-tenant customers a genuine security boundary between their MCP traffic and anyone else's, rather than a shared-process model where a bug in one tenant's parsing logic could leak into another's.

**Telemetry and packet inspection.** AXIOM parses JSON-RPC 2.0 frames in real time across both transports it needs to support: `stdio`, for the local-process MCP servers that dominate developer machines, and SSE, for the remote/hosted servers enterprises increasingly run behind their own infrastructure. Every `tools/list` response is hashed at first connection and re-verified on every subsequent session — directly closing the rug-pull gap, since a description that silently changes between sessions now triggers an alert instead of passing invisibly.

**State and memory storage layer.** A real-time session state machine tracks each agent-to-server conversation across its full lifecycle — connection, tool discovery, individual tool calls, and teardown — rather than treating each JSON-RPC frame as an isolated, stateless event the way a REST log would. On top of that state machine sits a FAISS-backed retrieval layer (a direct continuation of the same retrieval work already proven out in PRISM's AST-plus-FAISS knowledge graph) indexing embeddings of tool schemas, historical call patterns, and prior session context, so an auditor or the security engine itself can ask "has this tool's behavior or description drifted from what we saw last week" as a semantic similarity query instead of a manual diff.

**The security engine — mapped directly to the OWASP MCP Top 10.** OWASP published its first Top 10 dedicated specifically to MCP in 2025 (currently in beta, under project lead Vandana Verma Sehgal), and AXIOM's detection layer is built to cover it category by category rather than inventing a parallel, incompatible taxonomy:

| OWASP Category | What AXIOM Detects | How |
|---|---|---|
| MCP01 — Token Mismanagement & Secret Exposure | Hardcoded credentials, long-lived tokens, secrets surfacing in model memory or protocol logs | Automatic secret-pattern scanning across every intercepted frame; redaction before anything touches persistent storage |
| MCP03 — Tool Poisoning | Hidden instructions embedded in tool names, descriptions, or schemas | Real-time content inspection of every tool description against a maintained pattern library of injection/jailbreak phrasing, run at the proxy layer rather than only at install time |
| MCP04 — Software Supply Chain & Dependency Tampering | Compromised or typosquatted MCP server packages | Version pinning, checksum verification, and package-name similarity checks against known-good registries |
| MCP06 — Intent Flow Subversion | Prompt injection arriving via a tool description, a tool response, or a fetched document that hijacks the agent's reasoning mid-chain | Content inspection at every context-entry point, not just the initial tool list — the injection vector this project treats as the highest-frequency real-world attack |
| MCP07 — Insufficient Authentication & Authorization | Servers or tools that don't properly verify identity or enforce access boundaries | OAuth 2.1 scope auditing, flagging any token requesting broader access than its declared task requires |
| Audit & Logging Gaps | Blind spots where tool invocations go unrecorded | Full-parameter, full-context, timestamped logging of every call, structured for direct SIEM ingestion |

Beyond the explicitly numbered categories, AXIOM's detection layer also covers the framework's broader named risks: **shadow MCP server discovery** (surfacing unsanctioned servers running inside an org's network the same way a CASB surfaces shadow SaaS), **rug-pull / description-drift alerting** (the hashing mechanism described above), **tool-shadowing and cross-origin escalation** (flagging when one server's tool description references or attempts to modify another server's tools), the **confused-deputy pattern** (a server acting with its own broad privileges instead of the requesting user's actual scope), and **sandbox-escape containment guidance** for local servers (Landlock + seccomp + network namespace isolation on Linux, `sandbox-exec` on macOS).

## 1.3 Product Features & Developer Experience

**The Central Control Pane.** A unified dashboard giving a real-time, DevTools-style stream of every MCP message crossing the proxy, a visual dependency tree of which tool calls triggered which downstream calls (critical for multi-agent fan-out debugging), per-call latency and token-cost breakdowns, and a compliance log formatted for direct export into an enterprise's existing audit tooling.

**The Advanced Debugging Suite — "the Postman for MCP."** An interactive playground where a developer can mock a specific agent identity, isolate one MCP server's state independent of the others it's connected to, replay a failed tool call byte-for-byte to reproduce a bug, and live-test a schema change against real historical traffic before shipping it.

## 1.4 Market Positioning & Go-To-Market

**Customer profiles.** Two distinct tiers, deliberately sequenced: independent developers and small teams shipping agentic workflows fast, who need debugging visibility more than governance; and compliance-heavy mid-to-large enterprises running production AI workforces, who need the audit trail, SSO, and on-prem deployment options the free tier doesn't offer.

**GTM strategy.** A bottom-up developer-adoption loop, deliberately modeled on the distribution pattern that's already proven itself in this exact space — Invariant Labs' `mcp-scan` tool spread precisely because it's a single `uvx mcp-scan@latest` command with zero configuration, auto-discovering a developer's existing MCP setup across every major client. AXIOM's open-source CLI and local proxy follow the same zero-friction install pattern, then expand upward into paid team collaboration and, eventually, enterprise VPC/on-prem deployment once a company has enough developers already relying on it day to day.

**Competitive honesty.** This category is not empty, and pretending otherwise would be a planning error. `mcp-scan` already exists, is free, and is genuinely good at what it does: static, install-time and periodic scanning of a developer's configured MCP servers for known poisoning patterns. The NSA published its own MCP security design guidance in May 2026. OWASP's own framework is public. None of that is a reason to avoid this space — if anything, it confirms the problem is real and urgent enough that a federal intelligence agency felt compelled to weigh in within eighteen months of the protocol's release. AXIOM's differentiation has to be precise: it isn't a better static scanner, it's the **continuous, runtime, proxy-level layer** sitting above the scanners — the same relationship Datadog has to a server's raw log files. A team can run `mcp-scan` once a week and still have zero visibility into what actually crossed the wire in production yesterday. That gap is AXIOM's entire business.

# SECTION 2: MASTER BLUEPRINT FOR "VIGIL"
### Edge-Deployed Industrial Computer Vision Defect Detection

## 2.1 The Ultimate Thesis

**What Vigil is.** Vigil is an edge-native, end-to-end quality inspection system purpose-built for the small-to-midsize manufacturing units that the existing industrial-CV market structurally ignores — specifically the auto-ancillary and light-manufacturing units clustered across the Alwar–Bhiwadi–Neemrana industrial corridor, a segment defined by real operational conditions the enterprise vendors' product assumptions don't survive contact with.

**The industrial edge reality.** Covision, Lincode, SwitchOn, and Cognex have all proven computer-vision quality inspection works — some report false-scrap-rate reductions as high as 90%. But their go-to-market and product design both assume a large enterprise account: stable connectivity, controlled lighting, dedicated IT support, and a sales cycle measured in quarters. None of that describes a mid-size auto-parts unit in this corridor, where dust genuinely coats every surface, machinery vibration is constant, ambient light shifts by the hour as bay doors open and close, and internet connectivity cannot be assumed reliable enough to depend on for real-time cloud inference. A cloud-round-trip inspection system that adds even 200ms of latency waiting on a flaky connection is a system that gets switched off within a week on a fast-moving line.

**The data scarcity bottleneck — "the clean factory paradox."** The obvious approach to training a defect detector is collecting thousands of images of actual defective parts. The paradox: a well-run factory, by definition, doesn't produce many defective parts — that's what makes it well-run. The very facilities most capable of adopting new QC technology are the ones with the least raw material to train a traditional supervised object-detection model on.

## 2.2 The Complete Technical Architecture

**Hardware and edge deployment.** Global shutter cameras, specifically — not the rolling-shutter sensors common in consumer hardware, because rolling shutter introduces skew and distortion on anything moving at production-line speed, exactly the artifact that would corrupt a defect-detection model's input. Ring lighting rigs around each inspection point cancel out the ambient-light variability the facility can't control on its own. Inference runs locally on an NVIDIA Jetson-class edge module (Orin Nano/NX tier is the right balance of inference throughput to power draw and cost for this use case), housed in a ruggedized, dust-and-vibration-rated enclosure — the entire point being that the system keeps inspecting even when the plant's internet connection doesn't.

**Core vision stack.** A YOLO-family detector paired with a transformer-based segmentation head (SegFormer-class), implemented natively in PyTorch and compiled through TensorRT for the Jetson's hardware acceleration — this is a direct continuation of the object-detection and segmentation pipeline already built and proven in SPECTRA, repointed from satellite imagery classes to production-line defect classes.

**The synthetic data pipeline.** This is where the clean-factory paradox actually gets solved, and it's a direct reuse of skills already demonstrated: conditional generative models — the same family of conditional-GAN and diffusion-based image-to-image techniques used in SPECTRA's chrominance-prediction work — procedurally render realistic scratches, dents, micro-fractures, and structural misalignments directly onto a facility's own clean reference images. Instead of waiting months to accumulate real defect photographs, the model trains on a large, controllable, synthetically-augmented dataset from day one, with real defects folded in continuously as the no-code retraining loop below collects them.

**No-code retraining loop and SCADA integration.** A local, web-based management layer lets a plant's own quality engineer flag a new defect type and trigger a retraining pass without an ML engineer in the room — critical, because Vigil's actual customer will never have one on staff. On the industrial-integration side, Vigil speaks both of the protocols that actually matter in this environment: **Modbus**, the decades-old serial/TCP protocol still running the vast majority of legacy PLCs on Indian factory floors, and **OPC-UA**, the newer, more secure, more structured standard increasingly required for anything touching modern SCADA systems — supporting both is what makes Vigil installable on a fifteen-year-old line as easily as a new one.

## 2.3 Product Experience & Floor-Level UX

**Factory floor operator interface.** A zero-friction, low-latency local dashboard giving assembly-line workers real-time pass/fail classification and immediate visual (and where needed, audible) alerts — designed to be read at a glance by someone whose attention is on the line, not the screen.

**Plant management analytics portal.** A historical reporting layer tracking Overall Equipment Effectiveness (OEE — the standard manufacturing metric combining Availability × Performance × Quality) alongside batch-level quality trends and a direct financial accounting of losses prevented through reduced false-scrap rates, giving a plant owner a number they can put directly in front of their own management, not just a vague "quality improved" claim.

## 2.4 Market Positioning & Go-To-Market

**Customer profiles.** Small and mid-size manufacturers inside a specific, named regional industrial corridor — not "manufacturers in India," a targeting decision that is itself the strategy, not a limitation of it.

**GTM strategy.** High-touch, entirely local sales cycles: physical on-site proof-of-concept installations, a first-month ROI calculation delivered in person, and expansion driven through the same local industrial-zone relationships and word-of-mouth that got the first installation — a motion a remote, nationally-focused competitor cannot replicate at any price, because it isn't a product feature, it's a founder's physical presence.

# SECTION 3: ORGANIZATIONAL STRATEGY & FINANCIAL EXECUTION

## 3.1 Team Structure & Founding Archetypes

**Core team architecture.** Both ventures are built around the same lean, technically-dense founding profile: someone who has already operated inside competitive machine-learning environments under real evaluation pressure (national fraud-detection finals, top-percentile global ML competitions), delivered a complete production system inside a hackathon's compressed timeline, and shipped software that's actually collected paying users rather than sitting as a portfolio piece. The specific archetype needed is someone equally comfortable moving between low-level systems engineering (the FastAPI/Docker/proxy layer AXIOM needs) and physical edge infrastructure (the camera, lighting, and Jetson deployment Vigil needs) — a profile that is genuinely rare, and genuinely already demonstrated across the existing project history (Cronus's production infrastructure, PRISM's retrieval architecture, SPECTRA's vision pipeline).

**Complementary roles each venture needs beyond that core.** AXIOM needs a second technical hire or cofounder with real protocol/security depth — the gap between "can build a proxy" and "can be trusted by a compliance-heavy enterprise buyer to catch MCP01 through MCP07-class vulnerabilities" is a credibility gap as much as a technical one. Vigil needs an embedded/field engineer who can physically wire cameras into an existing production line and speak Modbus/OPC-UA to a legacy PLC, plus a locally-connected sales lead — ideally someone who already has standing relationships with manufacturing families in the target corridor, since this sale closes on trust built in person, not a demo call.

**Technical culture.** Both ventures are built on the same operating principle: ultra-rapid iteration against real deployed feedback (a live pilot line, a real developer's actual MCP traffic) over theoretical completeness, and full-stack ownership of a feature from architecture through to the interface a real user touches — the same mode of working already proven across every prior shipped project.

## 3.2 Comparative Financial & Operational Matrix

| Pillar | AXIOM | Vigil |
|---|---|---|
| **CapEx vs. OpEx** | Almost entirely OpEx — compute for the proxy/backend, no physical hardware to manufacture or install. Marginal cost per new customer is close to zero. | Real CapEx per installation — cameras, ring lighting, a Jetson module, and a ruggedized enclosure per inspection point, before any software revenue begins. Higher up-front cost, but it's also a harder-to-replicate physical footprint once installed. |
| **Scale Velocity** | Fast and geography-independent — an open-source CLI can be installed by a developer anywhere in the world in the time it takes to run one command. Global reach from day one. | Deliberately slow and local — each new customer requires a physical site visit and installation. Scale comes from depth in one corridor before geographic expansion, not breadth from day one. |
| **CAC vs. LTV** | Low CAC at the free/OSS tier (adoption is largely self-serve), rising sharply at the enterprise tier where a real sales motion is needed. LTV compounds through seat/usage expansion as a team's agent traffic grows. | High CAC per customer — a real, in-person sales cycle every time. LTV is anchored by hardware stickiness (a customer who has cameras and edge devices already installed and integrated with their PLC is very unlikely to churn to a competitor) and compounds further once Khata/other belt-strategy products are cross-sold into the same relationship. |
| **Long-Term Defensive Moat** | A knowledge moat — the security pattern library and historical traffic index get better with every customer's data, a classic data-network-effect flywheel, but one that a well-funded competitor with more traffic could eventually out-flywheel. | A relationship and physical-presence moat — genuinely difficult for a remote, better-funded competitor to buy quickly, because the moat is trust built on-site over months, not a feature that can be shipped in a sprint. |
| **Regulatory, Platform, and Macro Risk** | Platform risk is the dominant concern: general agent-observability platforms are already adding MCP-context tracing as a feature inside broader tools, and Anthropic/OWASP/NSA all now publish their own guidance directly — the risk is being absorbed as a checkbox feature of a larger platform rather than surviving as a standalone company. | Macro risk is the dominant concern: exposure to a single regional industrial economy, and slower absolute growth ceiling than a software-only play — but correspondingly lower platform risk, since no single incumbent can absorb "physical trust in this specific corridor" as a feature update. |

**The honest synthesis.** These two ventures sit at opposite ends of the same risk spectrum on purpose. AXIOM scales fast and cheap but is exposed to the exact kind of platform-absorption risk that has already closed off several adjacent categories in this research. Vigil scales slowly and expensively but sits on the one moat in this entire body of research that a funded, remote team genuinely cannot buy its way past. Running both in parallel isn't hedging for its own sake — it's deliberately pairing a fast, software-only bet against a slow, physically-defensible one, so that neither venture's specific failure mode is the whole portfolio's failure mode.
